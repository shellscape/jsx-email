## Migrating to Version 3.0.0

Version 3.0.0 of `jsx-email` modernizes the runtime, package exports, preview app, and CLI. It also removes the v1 compatibility escape hatch that was deprecated during the v2 cycle.

This guide was compiled from the `next/v3` branch history, the PRs merged into `next/v3`, and the v2 migration guide.

## Quick Checklist

- Use Node.js `22.0.0` or newer.
- Use React `19.1.0` and React DOM `19.1.0` or newer.
- Stop using CommonJS `require()` for `jsx-email` and `@jsx-email/*` plugins.
- Replace `Butan` usage with the current `Button` component.
- Use `create-mail` for new projects instead of the old create package.
- Review `Raw` and `Conditional` output if you snapshot rendered HTML.
- Review Tailwind usage against the v3 `Tailwind` component docs.

## Breaking Changes

### Node.js 22 Minimum

All v3 packages set the minimum Node runtime to `>=22.0.0`. Upgrade local development, CI, deployment images, and package manager toolchains before upgrading `jsx-email`.

```json
{
  "engines": {
    "node": ">=22.0.0"
  }
}
```

### React 19 Peer Dependencies

The v3 branch targets React 19. Projects should move to React and React DOM `^19.1.0`.

```sh
npm install jsx-email react@^19.1.0 react-dom@^19.1.0
```

TypeScript projects should also update React types to the matching v19 range.

### ESM-only Package Entrypoints

The v3 package output is ESM-only. The `jsx-email`, `jsx-email/compile`, `jsx-email/config`, and `@jsx-email/*` plugin export maps point at ESM artifacts and no longer provide CommonJS `require` mappings.

Before:

```js
const { render } = require('jsx-email');
const { compile } = require('jsx-email/compile');
```

After:

```js
import { render } from 'jsx-email';
import { compile } from 'jsx-email/compile';
```

If your project is still CommonJS, migrate the email build/render path to ESM or use dynamic `import()`.

### Plugin Packages Are ESM-only

The plugin packages are also ESM-only:

- `@jsx-email/plugin-inline`
- `@jsx-email/plugin-minify`
- `@jsx-email/plugin-pretty`

Before:

```js
const inline = require('@jsx-email/plugin-inline');
```

After:

```js
import inline from '@jsx-email/plugin-inline';
```

### `Butan` Has Been Removed

The v2 migration guide marked `Butan` as deprecated and stated that it would be removed in the next major version. In v3, `Butan` is no longer exported from `jsx-email`.

Before:

```tsx
import { Butan } from 'jsx-email';

<Butan href="https://example.com" style={{ padding: '10px 20px' }}>
  Click me
</Butan>;
```

After:

```tsx
import { Button } from 'jsx-email';

<Button
  href="https://example.com"
  width={160}
  height={44}
  backgroundColor="#111827"
  borderRadius={6}
  textColor="#ffffff"
>
  Click me
</Button>;
```

The current `Button` component is the supported path and is designed around explicit email-safe props rather than relying primarily on CSS styles.

### `create-mail` Replaces the Old Create Package

Use `create-mail` for new projects. The v3 scaffold targets Node 22 and React 19 and includes `react-dom` in generated project dependencies.

```sh
npx create-mail my-email-project
```

Existing generated projects should make sure they include:

- `react`
- `react-dom`
- `@types/react` when using TypeScript
- `typescript` when using TypeScript

### `Raw` and `Conditional` Rendering Changed Internally

The public component APIs remain the same, but the renderer now uses a rehype-based pipeline for `Raw` and `Conditional` output. This can change rendered HTML details and snapshots.

Review output if you rely on exact HTML strings. The v3 behavior includes:

- multiline `Raw` HTML emitted verbatim
- nested `Raw` inside `Conditional` preserved correctly
- safer conditional comment closing syntax
- improved placement for head-aware conditional content

## CLI Changes

### `--use-preview-props` Works for `build` and `check`

`email build` and `email check` both support `--use-preview-props`. When set, the CLI uses the template’s exported `previewProps` instead of separately supplied props.

```sh
email build ./emails --use-preview-props
email check ./emails/welcome.tsx --use-preview-props
```

### `check` Supports `--email-clients`

`email check` can limit compatibility validation to a comma-separated list of email clients.

```sh
email check ./emails/welcome.tsx --email-clients gmail,outlook
```

The v3 check command remains caniemail-based.

### Preview Build Path Behavior Was Aligned

Preview deploy builds were aligned with main-branch behavior. If you use `email preview --build-path`, verify CI/deploy scripts that assume a specific output layout.

```sh
email preview ./emails --build-path ./dist/email-preview
```

## Tailwind and Styling

The v3 `Tailwind` component uses UnoCSS for atomic CSS generation and supports Tailwind/Windi-style syntax. The docs call out that native Tailwind plugins are not supported; use UnoCSS presets and transformers instead.

If you were using the deprecated standalone `@jsx-email/tailwind` package, migrate to the `Tailwind` component exported by `jsx-email`.

```tsx
import { Tailwind } from 'jsx-email';
```

For Gmail compatibility and smaller output, prefer `production={true}` where appropriate.

These additions do not require migration, but they are new in v3 branch work and may replace custom code in your project:

- `QrCode`

## Package-level Notes

### `jsx-email`

- ESM-only export maps.
- Node minimum is `>=22.0.0`.
- React peer dependencies target React 19.
- `Butan` is removed.
- `Avatar`, `AvatarGroup`, and barcode components are exported.
- `jsx-email/compile` remains available, but through the ESM export path.

### `@jsx-email/plugin-inline`, `@jsx-email/plugin-minify`, `@jsx-email/plugin-pretty`

- ESM-only export maps.
- Node minimum is `>=22.0.0`.
- Peer dependency remains `jsx-email`.

### `create-mail`

- Use this package for scaffolding.
- Generated projects include React 19, React DOM 19, and the v3 runtime assumptions.

## Suggested Migration Order

1. Upgrade Node locally and in CI to Node 22 or newer.
2. Upgrade React and React DOM to the v19 range.
3. Convert any `require()` usage for `jsx-email` or plugins to ESM imports.
4. Replace `Butan` with `Button`.
5. Update scaffold/developer docs from the old create command to `create-mail`.
6. Run `email build`, `email preview`, and `email check`.
7. Compare rendered HTML snapshots for templates using `Raw`, `Conditional`, Tailwind-heavy markup, or buttons.
8. Use `email check --email-clients` for any client-specific validation you care about.
